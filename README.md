# Multi-Cloud IaC Platform

> Terraform modules that provision identical environments on AWS, GCP & Azure — VPCs, IAM, K8s clusters, and an observability stack — gated by GitHub Actions CI/CD.

## Architecture

```
┌──────────────────────────────────────────────────┐
│                 GitHub Actions CI/CD              │
│  terraform fmt → validate → plan → apply (merge) │
└────────────┬──────────────┬──────────────┬───────┘
             │              │              │
          ┌──▼──┐        ┌──▼──┐       ┌──▼──┐
          │ AWS │        │ GCP │       │Azure│
          │ EKS │        │ GKE │       │ AKS │
          └─────┘        └─────┘       └─────┘
```

## Tech Stack
| Layer | Tool |
|---|---|
| IaC | Terraform 1.7+ |
| Orchestration | Kubernetes 1.29 |
| CI/CD | GitHub Actions |
| Secret Management | HashiCorp Vault |
| Observability | Prometheus + Grafana |

## Getting Started
```bash
# Authenticate to all three clouds
aws configure
gcloud auth application-default login
az login

# Deploy dev environment on AWS
cd environments/dev
terraform init
terraform plan -var-file="aws.tfvars"
terraform apply -var-file="aws.tfvars"
```

## Key Design Decisions
- **Module reuse**: Each cloud has identical module interfaces — same inputs/outputs — making environment promotion predictable.
- **Remote state**: S3 / GCS / Azure Blob with state locking via DynamoDB / Firestore / Azure Table Storage.
- **Least privilege IAM**: All IAM roles generated by module with minimal permissions, no wildcards.
- **Cost tagging**: Every resource tagged with `env`, `project`, `owner` for chargeback reporting.

## CI/CD Pipeline
- PRs trigger `terraform plan` — output posted as PR comment
- Merges to `main` trigger `terraform apply` per environment
- Drift detection runs nightly via scheduled workflow
